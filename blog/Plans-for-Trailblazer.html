Plans for Trailblazer

The Idea

The idea first occured to me when one day I was tuning Roadrunner, a trajectory follower library that makes autonomous pathing very straightforward. I marveled at the different tuners I had to run, and thought to myself: what is it actually doing? what's the math behind it? That's when my trajectory-follower-research began. I started researching how these libraries worked, learned more about localization, dead wheels, and also a roadrunner alternative - Pedro Pathing (I still can't understand the name). I learned the main differences between Roadrunner and PedroPathing, which I will explain in a bit. But overall I concluded: it would be really cool if I could write my own library for this. Sure, the math's hard, but I'll just learn it. Not that hard right?...

Roadrunner vs PedroPathing

First, before I start writing any code, I gotta do some planning. The top 2 libraries used in FTC for path following is Roadrunner and PedroPathing. The goal of Roadrunner is to have consistent pathing.. The way roadrunner works, it computes all the motor velocities ahead of time and uses linear interpolation to determine what to set the motors at based on how far ahead it is in the path. It does also use PID in this process to stay on course, but I still don't really understand how they do this. Second is PedroPathing, which the good teams use for speed and accuracy. PedroPathing focuses on speed and accuracy along the path. How Pedro works: it creates a bunch of points along the line that it needs to follow. It uses drive, translational, rotational, and centipetal vectors to target the next closest point. This makes it to where it sticks to the path very well and can handle disturbances well. It is also very fast, which is why the majority of good teams use it. It does sacrifice on consistency though, since every run will be different by nature of the PID algorithm.

So, if I want to select a library to base mine off of, it would either have to:

- Compute the wheel velocities over time for the specified path

OR

- Generate a bunch of points and use PID to have the robot stick to the points.

On the face of it, the PedroPathing solution does seem to be the mathematically simpler option. Generating single points seems easier than generating full on velocities.

So ultimately I decided to go with this method. Having a fast, accurate path planner would be most ideal, especially if I want to end up using this on our actual FTC bot. It needs to, at the bare minimum, meet the times and speeds of PedroPathing. Now, the main reason I am doing this is for the learning experience. I have always wanted to have the most control over my code. Using other libraries just seems like cheating to me - after all, when using a library, are you really learning the fundamental skills? By actually applying your knowledge into a pretty big project like this, there is so much to learn.
